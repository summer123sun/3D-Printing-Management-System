## 〇、TL;DR（一分钟看完）

- **你们最大的风险**：3 人都从零学 Spring Boot，4 周内必须边学边干活
- **解决方案**：第 1 周全员**统一过一遍 Spring Boot 核心**（每人学一样的，最后能互相 review），第 2 周开始**专项开发 + 一边学一边写业务**
- **3 人分工**：D = 项目脚手架 + 鉴权 + 用户（基础设施型）；E = 打印任务 + 项目（核心业务型，对应前端 B）；F = 作品库 + 设备耗材 + 统计日志（辅助业务型，对应前端 C）
- **接口契约**：第 1 周末锁定 API 文档（前后端共用），前端先用 mock，第 3 周末开始真实联调
- **接口实现交付节奏**：登录/用户 第 2 周末；任务 第 3 周末；项目 第 4 周末；作品库/设备/耗材 第 5 周末；统计/日志 第 6 周末

## 一、团队现状评估 + 学习策略

### 1.1 你们现在的水平假设

| 已会 | 不会 |
|---|---|
| Java SE 基础（类、接口、集合、异常、IO） | Spring / Spring Boot 任何东西 |
| MySQL 基本 SQL | MyBatis / MyBatis-Plus |
| JDBC 概念 | Spring MVC / RESTful |
| 命令行 / IDE（IntelliJ IDEA） | Maven 多模块 |
| Git 基本操作 | JWT / 拦截器 / AOP |

### 1.2 学习策略（关键！）

> ❌ **错误做法**：3 个人各学各的、各做各的，最后代码风格不统一、互相看不懂
> ✅ **正确做法**：第 1 周 3 人**集体学** Spring Boot 核心，每人做同一个 demo，最后代码互相 review

**为什么**：
- Spring Boot 项目**一致性**比"谁技术最强"更重要——后端 3 人的代码要能互相接手
- 期末答辩老师可能会问"这段代码谁写的"——如果 3 人代码风格完全不一样，会被认为"只有 1 人干活"
- 联调时接口契约一致性比实现性能更重要——风格统一才能省沟通成本

### 1.3 第 1 周：3 人集体过一遍 Spring Boot 基础

**目标**：3 人各自能独立跑起一个 Spring Boot Hello World + 接一个 MySQL 库 + 写一个 RESTful 接口

**每天的任务**：

| 天 | 内容 | 产出 |
|---|---|---|
| Day 1 | IDEA + JDK 17 + Maven 环境搭建；spring initializr 生成项目；跑通 Hello World | 3 人各自能 `mvn spring-boot:run` |
| Day 2 | Spring Boot 自动配置原理（@SpringBootApplication、application.yml）；写一个 RESTful 接口返回 JSON | 3 人各自写 `/hello` 接口 |
| Day 3 | Spring MVC（@RestController、@RequestMapping、@PathVariable、@RequestParam）；接 MySQL + MyBatis | 3 人各自写 `/user/{id}` 返回一个 user 对象 |
| Day 4 | MyBatis-Plus（BaseMapper、内置 CRUD）；分页插件 | 3 人各自写 `/user/list?page=1&size=10` |
| Day 5 | **统一 demo**：3 人一起做一个"学生管理系统" mini demo（10 张表，能增删改查），互相 review 代码 | 风格统一、3 人代码 merge 不打架 |

**Day 5 的 mini demo 是关键**——它就是你们整个项目的"样板间"，所有约定（命名、包结构、统一返回、JWT）都在这里定下来。

### 1.4 第 2 周起：分头干 + 边写业务边学新技能

| 周 | 学什么新技能 | 干业务 |
|---|---|---|
| 第 2 周 | JWT（jjwt）、Spring Security（或手写拦截器）、CORS、统一返回、全局异常处理、Knife4j | D 写登录/用户；E 写任务模块 |
| 第 3 周 | 文件上传（multipart）、文件 IO；MyBatis 复杂查询（动态 SQL、resultMap） | D 写文件上传；E 写任务全流程；F 开始学 |
| 第 4 周 | 事务（@Transactional）、乐观锁、关联查询（@One/@Many） | E 写项目模块；F 写作品库 |
| 第 5 周 | 缓存（Redis，可选）；定时任务；AOP 日志 | F 写设备耗材 + 看板 |
| 第 6 周 | 性能调优（连接池、慢 SQL）；Knife4j 完善 | F 写日志；全栈联调 |
| 第 7 周 | 部署（Nginx + jar）；Docker（可选） | 部署文档 |
| 第 8 周 | 测试用例（Junit5）；压力测试（JMeter，可选） | 演示准备 |

## 三、3 人后端分工

> **关键原则**：D 负责**基础设施型**（其他人都要用的东西），E 负责**核心业务**（对应前端 B，你也是前端 B），F 负责**辅助业务**（对应前端 C）。

| 人员 | 负责模块 | 工作量占比 | 关键交付 |
|---|---|---|---|
| **D** | 项目脚手架 + 鉴权 + 用户 + 文件上传 + 通用组件 | ~30% | 启动类、统一返回、JWT、CORS、Knife4j、登录 CRUD、用户 CRUD、文件上传 |
| **E**（你，前端 B 也是你） | 打印任务 + 项目（核心业务） | ~40% | 任务全流程 + 项目全生命周期 |
| **F** | 作品库 + 设备 + 耗材 + 统计 + 日志（辅助业务） | ~30% | 看板数据 + AOP 日志 + 库存预警 |

## 五、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|---|---|---|---|
| 3 人学习速度不一致 | 高 | 高 | 第 1 周全员一起学，互相帮 |
| D 进度慢拖累 E 和 F | 中 | 高 | D 第 2 周必须交付项目脚手架，否则 E/F 起不来 |
| 接口契约反复修改 | 中 | 中 | 第 1 周末锁定，第 2 周后改要全组同意 |
| 文件上传路径前后端不一致 | 中 | 中 | D 第 3 周出统一规范 |
| 事务边界没考虑清楚 | 中 | 中 | E 第 3-4 周重点关注，每个写操作想清楚"哪些 SQL 要在同一事务" |
| 联调时接口对不上 | 高 | 高 | 第 3 周末是硬节点，前端 E 必须出 mock 跑通流程 |
| 答辩时被问代码细节 | 中 | 高 | 3 人必须互相 review 第 7 周，至少要能讲清楚对方的核心代码 |
