## 〇、v2 版本变更说明（先看这里）

v1 文档里有些东西冗余、有些不实用、还有前后矛盾——v2 一并处理。先列清楚改了什么，方便你 review。

### 0.1 ⚠️ 关键技术栈矛盾修正

**v1 原文**："后端采用 Java Swing、JDBC、IO 或网络编程等"

**问题**：Java Swing 是**桌面 GUI 组件库**（做窗口程序的），它**做不了 Web 后端**。你同时说"前后端分离 + Vue3 前端"，那后端必须能响应 HTTP 请求给浏览器——Swing 干不了。

**v2 修正**：后端栈改为 **Java Spring Boot + MyBatis + Spring MVC + 内嵌 Tomcat + RESTful API + 文件 IO + JWT 鉴权**。这是 Java 生态最主流、最成熟的 B/S 后端组合，跟 Vue3 是黄金搭档。

> **如果你坚持要用 Swing**：那这是 C/S 架构（桌面客户端 + Socket 服务端），Vue3 前端整个就作废了——这是两个完全不同的产品，请告诉我再调整。

## 一、系统概述

### 1.1 项目背景

3D打印科创会是一个新成立的学生社团，主要开展3D打印、3D建模的教学与科普活动。社团日常运营中面临以下痛点：

- **打印任务管理混乱**：社员想打印作品时，缺乏统一的申请和排队机制，经常出现多人同时想用打印机、参数设置错误导致耗材浪费等问题
- **知识传承困难**：学长毕业后的建模经验、调机技巧没有系统记录，新社员重复踩坑
- **项目协作不透明**：多人合作一个作品时，不清楚谁负责建模、谁负责打印、当前进度到哪里
- **设备耗材管理粗放**：不知道打印机累计使用时长、耗材库存剩余量，经常出现打印到一半没料的情况

### 1.2 系统目标

开发一套面向3D打印科创会的内部管理系统，实现：

1. 打印任务在线申请、技术审批、排队执行、取件签到全流程
2. 社团项目创建、成员分工、进度跟踪、成果归档
3. 设备维护与耗材库存预警
4. 作品库积累与知识传承（**重点是心得总结，不是点赞**）
5. 不同角色权限控制

### 1.3 用户角色

| 角色 | 权限 | 模拟人数 |
|---|---|---|
| 社长 | 全部功能，包括成员管理、审批、统计看板 | 1 人 |
| 技术骨干 | 审批打印任务、维护设备、创建项目 | 3 人 |
| 普通社员 | 提交打印任务、参与项目、浏览作品库 | 15 人 |
| 新成员 | 提交打印任务（需审批）、浏览作品库 | 10 人 |

#### 1.4.2 后端技术栈

> ⚠️ **v2 修正**：用 **Spring Boot** 替代原 v1 提到的"Swing"

| 类别 | 技术 | 用途 |
|---|---|---|
| 框架 | Spring Boot 3.x | 核心框架，快速启动 |
| Web | Spring MVC | RESTful API |
| ORM | MyBatis / MyBatis-Plus | SQL 映射，比纯 JDBC 省事但可控 |
| 数据库 | MySQL 8.0+ | 数据持久化 |
| 连接池 | HikariCP | Spring Boot 默认 |
| 鉴权 | JWT (jjwt 0.12.x) | 前后端分离的 token 鉴权 |
| 文件存储 | 本地磁盘 + 路径规范 | STL/图片/PDF 上传 |
| 工具 | Lombok | 简化 Java Bean |
| API 文档 | Swagger/Knife4j | 接口文档自动生成 |
| 构建 | Maven 3.8+ | 依赖管理 |
| JDK | JDK 17 LTS | （不要用 JDK 8，新项目没必要） |

**用到的 Java 技术点**：
- **JDBC**（MyBatis 底层就是 JDBC）
- **IO / NIO**（文件上传下载、STL 文件读取）
- **网络编程**（HTTP 服务、Tomcat 内嵌）
- **多线程**（可选：文件分片上传用线程池）
- **集合 / 泛型 / 反射**（框架底层会用到）

> 如果学校课程要求必须用"纯 JDBC 不能用 MyBatis"，可以把 MyBatis 换成 Spring 的 `JdbcTemplate` 或原生 `Connection`，业务层代码调整量不大。
